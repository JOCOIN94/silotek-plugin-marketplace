# SB-STM SERIAL_CHAR_DELAY=100 검증 세션 요약 (2026-06-12, COM13)

- **목적**: r0에서 확인된 잔존 유실(10ms 지연에서도 VCFGINDOOR→VFIDOR, 10자 중 4 유실)이 100ms 지연으로 해소되는지 검증. 설정은 설치 플러그인 캐시 `plugin.json`에 `SERIAL_CHAR_DELAY=100` 임시 주입.
- **결과**: **유실 0 — 전 회차 통과.** VCFGINDOOR 5연속 + HELP 2회 + VRGCNT 1회, 송신 에코 총 64자 전부 온전(`Input : X` 대조), 다중행 응답도 매회 완전 회수.
- **시험 조건**: 장비 v2.19, 대기 상태(과금 동작 아님). PC 재부팅·케이블 정리 직후 세션. 운영(과금) 중 트래픽 상황은 미시험 — 어차피 운영 중 STM 콘솔 쓰기는 금지 규칙(notes-uart1-sharing.md §6)이므로 시험 범위 밖.
- **한계**: UART1 공유 경쟁 자체는 지연으로 사라지지 않는다(notes-uart1-sharing.md §3) — 100ms는 "3바이트 누적 조건이 사실상 안 걸리게" 만드는 운영 완화책이며, 근본 해결은 펌웨어 수정안 A(보류 중).
- **부수 교훈 — 물리 미연결의 침묵**: DEBUG_STM32 핀이 빠진 상태에서도 USB 어댑터 포트는 정상 열거·열림·write 성공으로 보인다(UART엔 링크 감지가 없음). 16:40 송신이 에코 0으로 끝난 원인이 이것. **포트 보임 ≠ 보드 연결, 에코만이 수신의 증거** — 에코 대조 규칙의 근거가 하나 더 늘었다.
- **후속 결정 필요**: 100ms 표준화 방식 — 서버 기본값 변경 vs 배포 플러그인 env 고정 vs 문서 안내. (임시 주입은 플러그인 1.2.0 업데이트 시 소멸.)
