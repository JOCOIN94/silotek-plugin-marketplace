# SB-STM UART1 공유 충돌 — 문자 유실 근본 원인 조사

조사: Codex (소스 전수 추적, 2026-06-12) · 검증·보정: Claude (소스 표본 대조)
대상: `smartBay/Core/Src/main.c` @ `0b01761` (v2.34)
배경: PC 콘솔(DEBUG_STM32 헤더, COM13)에서 문자 간 10ms 지연 전송에도 한 글자 걸러 유실(실측 `VCFGINDOOR`→`VFIDOR`, exploration/2026-06-12-r0 참조).

## 판정: 가설 확정 — UART1을 CLI와 ESP 프로토콜이 공유하며, 프로토콜 스캐너가 CLI 바이트를 소비한다

| 위치 | 함수 | 컨텍스트 |

| main.c:2284 | `Receive_Wait_Byte()` | 프로토콜 핸들러의 소비 경로 (검증됨 — Rx1RdCounter 전진) |

### 2. 디스패치 구조 (메인 루프 내 실행 순서)

```
main loop
 ├─ Command_Proc()  (main.c:17844 — CLI보다 먼저)
 │    └─ ChkPacket() (main.c:8623 호출, 본체 7856)
 │         ├─ buf[0]==0xFF        → 1바이트 소비 + 0xFF 에코 (핑 응답, 7865-7868)
 │         ├─ buf[0..1]==0xFA,0xFA → 장프레임 소비 루프 (7887~)
 │         ├─ buf[2]==0xFB        → 3바이트 프레임 [CMD,VAL,0xFB] 처리 (8065~)
 │         └─ 그 외(비프레임)      → **Receive_Wait_Byte로 1바이트 소비 후 리턴 (8060-8064)**
 └─ CLI 파서 (main.c:18157~) — Receive_NoWait_Byte로 소비
```

### 3. 유실 메커니즘 (실측 패턴 완전 설명)

- `ChkPacket()`은 버퍼에 **3바이트 이상** 쌓였을 때 스캔한다(7875-7883).
- 우리 ASCII 명령 바이트가 3개 이상 누적된 시점에 `Command_Proc()`이 먼저 돌면, 프레임 패턴이 아니므로 skip 분기(8062)가 **1바이트를 버린다**. 다음 패스에서 CLI가 1바이트를 읽는다 → **한 글자 걸러 유실**.
- 루프 타이밍에 따라 CLI가 먼저 다 읽어가면 전체 도달 → 재시도가 확률적으로 성공하는 이유.
- 문자 간 지연을 늘려도 두 소비자 경쟁 자체는 사라지지 않는다(3바이트 누적 조건이 덜 걸릴 뿐).

### 4. 역방향 위험 (콘솔 → 보드 간 프로토콜 오염)

위험 실재. ASCII 시퀀스가 우연히 프레임 조건(특히 buf[2]==0xFB는 ASCII로 불가능하나 0xFA·0xFF는 수신 노이즈와 조합 가능)을 만족시키거나, ESP 프레임 수신 도중 ASCII가 끼어들면 체크섬 불일치·상태머신 오진행을 유발할 수 있다. **운영 중 SB-STM 콘솔로의 쓰기는 보드 간 통신에 간섭하는 행위**로 취급할 것. (참고: ESP→STM 명령 송신은 `Send_MsgToESP32` 계열이 UART3을 쓰는 경로도 있어, UART1/UART3 역할 분담의 전체 그림은 수정 설계 시 재확인 필요.)

### 5. 수정안 (미구현 — 설계 검토용)

| 안 | 내용 | 규모 | 비고 |
|---|---|---|---|
| **A. 바이트 클래스 디멀티플렉스 (권장)** | skip 분기(8060-8064)를 보정: 비프레임이고 `buf[0]`이 printable ASCII(0x20-0x7E)면 **소비하지 않고** 리턴(CLI 몫으로 남김). 비프레임 + 비인쇄 바이트만 기존대로 1바이트 소비(프로토콜 재동기 보존). 3바이트 프레임 종결자 0xFB는 비인쇄라 ASCII 명령과 충돌 없음 | 소형 (분기 1개 수정) | 재플래시 필요. Codex 원안의 "0xFB 헤더" 표기는 부정확 — 0xFB는 3바이트 프레임의 **종결자**이고 헤더는 0xFF/0xFA(Claude 보정) |
| B. 콘솔 UART 분리 | CLI·printf를 UART4로 이관 (`MX_UART4_Init`/`huart4` 심볼 존재 확인됨) | 중형 | UART4 핀의 보드 노출 여부(하드웨어) 확인 필요. 근본 분리 |
| C. idle 윈도 게이팅 | 프로토콜 무트래픽 구간에만 CLI 수용 | 중형 | 완화책 — 근본 해결 아님 |

**권장 경로**: A를 1차 적용(최소 변경·즉효) → 검증 후 차기 보드 리비전에서 B(물리 분리) 검토.

### 6. 적용 전까지의 운영 규칙 (아틀라스·스킬 반영)

1. SB-STM 콘솔 쓰기는 에코(`Input : X`)를 송신문과 대조해 유실 검출 → `Unknown command` 확인 후 재시도.
2. 운영(과금 동작) 중에는 SB-STM 콘솔 쓰기를 피한다 — 프로토콜 오염 위험.
3. 빈 줄 전송 금지(빈 줄 = NVIC_SystemReset).
4. 읽기 관찰(STM TX 방향)은 무해.
